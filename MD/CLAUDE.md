## Key Game Mechanics (quick reference)

- **Corporation groups**: Group 1 (NYH, NYC, CP) → Group 2 (CPR, UP, ATS, SP) → Group 3 (NP, CN, TP, ORN, WP, GMO); each group unlocks when all of the previous group have floated
- **Bonds (Schuldschein)**: one per corp; director cannot sell while bond outstanding; outstanding bonds at game end are director's personal liability
- **Monopoly fee**: owning >60% costs 20% of face value per share above threshold
- **E-trains**: express variant of each base type; pay N stops, visit unlimited
- **Salt Lake City bonus**: $30 normally; reduced to $15 while SOC (P7) is open
- **Private close triggers**: SOC closes when CPR or UP floats; NHSC closes when NYH floats; PSC closes when WP pays first dividend; FNY closes when NYC pays first dividend
- **Golden Spike event**: triggered when SLC transcontinental bonus first collected

## Session Workflow

### 1 — Orient

Read CLAUDE.md (auto) + `MD/inwork.md`. Then present the in-work items grouped by readiness:

- `[t]` in testing — **resume immediately**: re-run 6a (IRB) to confirm still green, then continue browser testing
- `[>]` needs PR — submit before starting new work
- `[~]` partial — active development; suggest by proximity to completion
- Quick wins (L1, well-scoped, no blockers) highlighted

Ask user to select one item before proceeding.

### 1b — Check open PRs

Check for open upstream PRs and surface any unresolved review comments:

```bash
gh pr list --repo tobymao/18xx --author @me --state open
```

For each open PR, pull inline comments:

```bash
gh api repos/tobymao/18xx/pulls/<N>/comments \
  --jq '[.[] | {author: .user.login, path: .path, line: .original_line, body: .body}]'
```

For each comment, classify and recommend:

| Label | Meaning |
|-------|---------|
| **Fix now** | Wrong behavior, wrong syntax, visual bug, or easy one-liner — address before any new feature work |
| **Defer (agreed)** | Reviewer accepted a later-PR deferral — skip |
| **Discuss** | Ambiguous; needs a rule decision or design choice — raise with user |

Apply all **Fix now** items on the PR branch, then push and commit: `fix(1862gs): address PR review — <short description>`.

### 2 — Branch

Each item in `MD/inwork.md` ends with a backtick-wrapped branch name. Three cases:

- **Named branch** (e.g. `` `1862_GS` ``) → check it out directly, skip branch creation
- **`?`** → branch unknown; ask user before proceeding
- **No tag** → new item; create a branch

For new branches, check divergence first:
```bash
git -C ~/18xx/18xx fetch origin
git -C ~/18xx/18xx log --oneline origin/master..origin/1862_GS | head -20
```
State the recommended base and reason, then create:
```bash
git -C ~/18xx/18xx checkout -b 1862_featurename <recommended-base>
```
After creating, add the branch tag to the item in `MD/inwork.md`.

### 3 — Discover

Read the relevant rules section (PDFs in `~/18xx/`). Read the matching `htmldoku/` page from the Documentation Lookup table. Grep the engine for existing implementations of the same mechanic.

### 4 — Consult until clear

Work with the user to surface every edge case, rule interaction, and ambiguity. Do not start coding until both parties have confirmed the picture is complete.

End the consulting phase with a **sign-off card**:

```
Rules cited:         §X.Y p.NN — <summary>
Edge cases:          <list every one explicitly>
Interactions:        <which other features are affected and how>
Ambiguities:         <each one, with resolution and rule citation>
Layer:               L1 / L2 / L3
Comparator title:    <title read>
Test scenarios:      <written here — see below>
Proceed? ✓ / ✗
```

Each test scenario:
```
Scenario:     <game state prerequisite>
Action:       <what the player does>
Expect:       <exact outcome>
Test method:  IRB | Browser | Both
IRB setup:
  require_relative 'lib/engine' unless defined?(Engine)
  g = Engine::Game::G1862UsaCanada::Game.new(%w[Alice Bob Charlie])
  # direct state setup (phase, trains, cash, entities)
  puts <result_expr> == <expected> ? "PASS" : "FAIL: got #{<result_expr>}"
```

| Method | Use when |
|--------|----------|
| IRB | Logic: revenue, phase trigger, share movement, cash, train rules |
| Browser | UI: tile rendering, route highlight, step sequencing, token placement |
| Both | Logic component testable in IRB + visual confirmation needed in browser |

Omit the IRB block for Browser-only scenarios. One scenario per edge case. Do not proceed until user confirms the card.

### 5 — Implement

One task per session. If an adjacent bug surfaces, log it in `MD/bugs.md` and finish the current task first. Follow coding guidelines (`~/18xx-docs/htmldoku/coding-guidelines.md`).

### 6 — Test

**6a — Automated (IRB, internal):** Run every IRB snippet from the sign-off card autonomously:

```bash
docker compose exec rack irb -r ./lib/engine
```

Fix failures silently until all pass. Then update item status in `MD/inwork.md`:
- All scenarios IRB-only → set `[>]` (ready for PR), skip 6b
- Any Browser/Both scenarios → set `[t]` (testing in progress)

**6b — Manual (Browser), only after all IRB pass:**

> "All automated tests passed. N scenario(s) require browser verification:"

For each: describe what to open, what to click, what to look for.

- User reports **"test XY passed"** → Claude commits `test(1862gs): <scenario name> ✓`
- Test fails → fix code → re-run IRB → re-present browser scenario → commit when passing

When all browser scenarios pass, update item to `[>]` in `MD/inwork.md`.

Do not move the item to done until every scenario has a passing commit.

### 7 — Wrap up

All tests committed:

- Move item from `MD/inwork.md` to `MD/done.md`
- Update internal tracking: `MD/bugs.md` · `MD/decisions.md` as needed
- Ask: **"Does the documentation need updating?"** → edit `htmldoku/*.md`, run `generate-html.py`, commit both source and rendered HTML
- Commit all `~/18xx-docs/` changes together
